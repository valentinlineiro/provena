# v0.3.0 — Projection proof

**Date:** 2026-07-23
**Status:** Approved design
**Version:** 0.3.0

## Objective

Establish the architectural contracts that make Provena destination-independent.

The implementation SHALL satisfy:

- **Identity is immutable.** No projector mutates the source.
- **Representations are derived from Identity.** No representation is a direct
  edit target; all are projections from the canonical source.
- **Renderers never change representations.** A renderer is a pure
  serialization step.
- **Multiple representations preserve the same meaning.** A name, a date, a
  claim — same source, same value, different shapes.
- **Multiple renderers may serialize the same representation.** One `ResumeModel`
  feeds both Markdown and HTML without re-projecting Identity.

## Current state

```
Profile → toResumeProjection() → ResumeProjection → MarkdownResumeRenderer.render() → string
```

`toResumeProjection` is a projector with no formal contract. There is only one
destination (Markdown), so destination-independence cannot be tested.

## Design

### Representation vs artifact

A **representation** is part of the domain model — a value object shaped for
a context (résumé, JSON Resume, LinkedIn). It is not a file format. An
**artifact** is an external serialization of a representation (`.md` file, HTTP
response body, CLI stdout). A representation can produce many artifacts; an
artifact never back-propagates into the domain.

### Pipeline

```
Profile
   │
   ▼
Projector<TModel>      (pure: Profile → TModel)
   │
   ▼
TModel                  (representation — domain-shaped, not format-shaped)
   │
   ▼
Renderer<TModel>       (pure: TModel → string)
   │
   ▼
string                  (artifact)
```

### Contract

```ts
interface Projector<TModel> {
  project(profile: Profile): TModel
}
```

A pure function. No effects. No access to Renderer. The existing projection
functions become implementations of this contract.

### Naming

| Term | Role | Example |
|------|------|---------|
| `TModel` | Intermediate representation, domain-shaped | `ResumeModel`, `JsonResumeModel` |
| `Projector<TModel>` | Transforms Profile into a representation | `ResumeProjector` |
| `Renderer<TModel>` | Serializes a representation to string | `MarkdownResumeRenderer` |

`JsonResumeModel` (not `JsonResume`) — keeps the distinction clear that this
is a domain representation, not the JSON artifact.

`ResumeModel` replaces the former `ResumeProjection` type.

### Packages

```
packages/
  core/         + Projector<T> interface
  yaml/           unchanged
  markdown/       Renderer<ResumeModel>
  jsonresume/     Projector<JsonResumeModel> + Renderer<JsonResumeModel>
  cli/            uses projector → renderer chain
  demo/           uses projector → renderer chain
```

`@provena/jsonresume` is a new package implementing both a `JsonResumeProjector`
and a `JsonResumeRenderer` (produces JSON string).

### Tests

Three invariants, each in its own test:

1. **Identity unchanged after projection**
   ```ts
   const profile = loadProfile()
   resumeProjector.project(profile)
   jsonResumeProjector.project(profile)
   expect(profile).toBeUnchanged()
   ```
   (exists already for Resume; extends to JSON Resume)

2. **One representation, multiple renderers**
   ```ts
   const representation = resumeProjector.project(profile)
   const md = markdownRenderer.render(representation)
   const html = htmlRenderer.render(representation)
   expect(representation).toEqual(resumeProjector.project(profile))
   // md !== html, both contain the same name
   ```

3. **Two distinct representations preserve the same meaning**
   ```ts
   const resume = resumeProjector.project(profile)
   const jsonResume = jsonResumeProjector.project(profile)
   expect(resume.name).toBe(profile.identity.person.name)
   expect(jsonResume.basics.name).toBe(profile.identity.person.name)
   ```

### What does NOT change

- Entity types, Identity, Profile schema
- Validation logic
- YamlWorkspaceLoader
- WorkspaceLoader interface
- package dependency direction (plugins depend on core, never vice versa)

## Architectural invariants

The following invariants govern all code in this version. Every test exists
to verify one of them; every regression is a violated invariant.

| Id | Invariant |
|----|-----------|
| I1 | Identity is authoritative. |
| I2 | Projectors never mutate Identity. |
| I3 | Renderers never mutate representations. |
| I4 | Representations are deterministic. |
| I5 | Equivalent meaning may have multiple representations. |
| I6 | The same representation may have multiple artifacts. | **Deferred** — the architecture enables this invariant, but it will become executable when a second `Renderer<ResumeModel>` exists (e.g. HTML). |

## Naming convention

Provena uses the following pattern for all destination packages:

```
<Target>Projector     → produces <Target>Model
<Target>Renderer      → consumes <Target>Model
```

Where `<Target>` is the consumer context (Resume, JsonResume, LinkedIn, etc.),
not the output format. A `ResumeProjector` does not project toward Markdown;
it projects toward a `ResumeModel`, which any `Renderer<ResumeModel>` may
serialize. This convention is fixed and must be consistent across all packages.

## Future possibility (not for v0.3)

```ts
interface Projector<TSource, TModel> {
  project(source: TSource): TModel
}
```

Generalization when a second source type exists. Not before.

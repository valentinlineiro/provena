# v0.4.0 — Adoption

**Date:** 2026-07-23
**Status:** Draft
**Version:** 0.4.0

## Objective

From architecture to adoption.

A developer who discovers Provena today can, in under five minutes:

1. Install it
2. Understand what it solves
3. Create a workspace
4. Run `provena render . --format jsonresume`
5. Get a useful result without reading architecture docs

## Acceptance criteria

- **Quick Start**: one page, end to end, every command executable verbatim.
- **CLI**: `render`, `validate`, `--help`, clear errors, actionable messages.
- **Example**: a minimal workspace that works without modifications.
- **API**: `@provena/core` exports are documented and stable.
- **Installation**: no unexpected steps or unnecessary configuration.

## Design principle

Every step in the Quick Start must correspond exactly to a command the user
can execute. If the README says:

```bash
npm install && provena render . --format jsonresume
```

that command must work as written. No footnotes, no hidden prerequisites, no
"before you start" clauses. The documentation IS the executable
specification of the adoption experience.

## Out of scope

- New identity types or domain entities
- Model complexity increases
- Internal optimizations
- Additional renderers (unless they clearly improve onboarding)
- I6 — deferred until a real second `Renderer<ResumeModel>` exists

## Scope

### CLI

Current state: `provena render <workspace>` — hardcoded to Markdown, no help,
minimal error messages.

Target:

```
provena render <workspace>              # default: markdown
provena render <workspace> --format jsonresume
provena validate <workspace>
provena --help
provena render --help
```

Requirements:
- `render` subcommand accepts `--format` with values `markdown` | `jsonresume`
- `validate` subcommand loads + validates workspace, exits 0/1
- `--help` shows usage, subcommands, format options
- Error messages include the actual problem and what to do
- Non-zero exit code on failure
- Output file is named `resume.<ext>` in the workspace directory (`resume.md`, `resume.json`)
- Only one format per invocation; run twice for both

### Public API surface

Document what `@provena/core` exports and which exports are stable.

| Export | Kind | Stable | Notes |
|--------|------|--------|-------|
| `Identity` | type | yes | Domain aggregate |
| `Profile` | type | yes | Canonical model |
| `Projector<TModel>` | interface | yes | Contract for projections |
| `Renderer<TModel>` | interface | yes | Contract for renderers |
| `WorkspaceLoader` | interface | yes | Contract for data sources |
| `validate()` | function | yes | Validation logic |
| `resumeProjector` | object | yes | Produces `ResumeModel` |
| `ResumeModel` | type | yes | Resume representation |
| `LinkedInModel` | type | no | Unstable, no renderer |

### Installation

Current: `npm install` in repo root. Need `npx` support eventually, but for
v0.4 the path is `git clone + npm install + npm run provena --`. The goal is
to document this clearly rather than hide it.
